# modules/nixos — Switchboard modules

Every file here follows the **Switchboard** pattern. A module declares options under `my.*`, defaults them to `enable = false`, and only activates its `config` when a host opts in.

## Module template

```nix
{ lib, config, pkgs, ... }:
let
  cfg = config.my.services.foo;
in
{
  options.my.services.foo = {
    enable = lib.mkEnableOption "Foo service";
    # extra options here
  };

  config = lib.mkIf cfg.enable {
    # actual configuration — services, packages, systemd units, etc.
  };
}
```

## Adding a new module — checklist

1. Create `services/<name>.nix` (or top-level `<name>.nix` for cross-cutting concerns).
2. Add the import to `default.nix` — **without this, the module is invisible to hosts.**
3. Declare options under `my.<area>.<name>` (containers go under `my.containers.*` in `nix-presets`).
4. From `<repo-root>` run `just maintenance::sync-agent` to refresh `../docs/OPTIONS.md`.
5. Opt in from a host file: `my.<area>.<name>.enable = true;`.
6. Verify: `just maintenance::check`.

## Blast-radius lookup

Two ways:
- `just maintenance::impact <path>` — fast, prints consuming hosts directly.
- Grep `../docs/OPTIONS.md` for the namespace (e.g. `my.desktop`) — same data, browsable.

The index is regenerated by `sync-agent`.

## Don't

- Don't declare options outside the `my.*` namespace.
- Don't put `lib.mkIf` checks against `cfg.enable` outside this file's `config` — they belong in this module.
- Don't add `enable = true` defaults — hosts must opt in.
